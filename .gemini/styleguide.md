# styleguide.md

## 基本方針

丁寧で親しみやすい日本語でレビューする。 良い設計や改善意図がある場合はそれを簡潔に認めつつ、 問題点は曖昧にせず、具体的かつ率直に指摘する。 重大な問題（バグ、セキュリティ、壊れる変更、運用事故につながる変更）は、影響と修正方針が分かるよう明確に伝える。 可読性と実務上の有用性を優先する。 アクセシビリティへの対応は3年後に予定されているため、現在は言及しない。


### Path: **/*.py

## レビューの優先順位
- 指摘の優先順位は「正しさ > セキュリティ > 壊れやすさ > 保守性 > 性能 > スタイル」の順にする
- 単なる好みではなく、再現しうる不具合や保守コストの根拠があるときに指摘する
- 軽微なスタイル差分のみなら重大な指摘として扱わず、単なる提案に留める

## 型・データ構造
- 関数・メソッドの引数と戻り値には原則として型ヒントを付ける
- `Any` は極力避け、必要なら具体型・Protocol・TypeAlias・Generics を検討する
- `Optional[T]` / `Union[A, B]` より Python 3.10+ の `T | None`, `A | B` を優先する
- 外部入出力、永続化境界、バリデーションが必要な構造化データには `pydantic.BaseModel` を優先して検討する
- 内部表現では用途に応じて `pydantic.dataclasses` / `TypedDict` / 明確なクラス設計も許容し、何でも `dict` に押し込まない
- 既存の Pydantic ベース設計と整合する場合は、内部表現でも `pydantic.dataclasses` を優先して検討する

## 命名
- 変数名・関数名は意味が明確なものを使う
- 不必要な略語は避ける
- bool は `is_`, `has_`, `can_`, `should_` など意図が分かる名前を優先する

## 制御フロー・構造
- 深いネストは避け、早期 return / continue / guard clause で読みやすくする
- 1 関数 1 責務を意識し、長すぎる関数は分割を検討する
- マジックナンバーやハードコード文字列は、意味がある場合は定数化を検討する
- 単純な変換やフィルタは内包表記を優先する

## 例外処理・ログ
- `except Exception` のような広すぎる捕捉は避ける
- 必要な場合は `except (ValueError, KeyError):` のように具体的な例外を扱う
- 例外を握りつぶさず、適切にログ出力するか再送出する
- 根本原因を直さず try-except で隠している場合は強く指摘する
- ログには個人情報・認証情報・機密値を出力しない

## docstring・コメント
- 公開関数や意図が読み取りにくい関数には docstring を推奨する
- コメントは「何をしているか」より「なぜそうしているか」を優先する
- 複雑な条件分岐、境界条件、回避策には理由が分かるコメントがあるか確認する

## テスト
- バグ修正には再発防止のテスト追加を検討する
- 境界値、異常系、None/空配列/空文字などの扱い漏れがないか確認する

## セキュリティ
- SQL は文字列連結ではなくパラメータバインディングを使う
- ユーザー入力は検証・正規化する
- ファイルパス、URL、コマンド実行、デシリアライズは危険入力として扱う
- 秘密情報やトークンをコード・ログ・例外に露出しない

## ディレクトリ構成 / コロケーション
- API Routes (`route.ts`) や Server Actions、ロジックは機能単位でのコロケーションを優先する
- 早すぎる共通化は避け、軽微な重複よりも責務の近さと変更容易性を重視する

## import ルール
- `@/app/...` 形式の絶対パス import は避ける
- `@/lib/...` と `@/components/...` は、全体で共有すべき安定したリソースに限定して使う
- 認証・セッションなど横断的責務は例外として許容する

## 型安全性
- `any` は使わない
- 不確実な値には `unknown` を使い、型ガード・Zod 等のバリデーション・narrowing で安全に絞り込む
- 関数の引数・戻り値が曖昧な場合は明示的な型定義を検討する
- `interface` と `type` は用途に応じて選び、型の責務が分かる名前にする
- nullable な値は放置せず、`null` / `undefined` の分岐を明確に扱う

## 設計・保守性
- 暗黙の型変換や truthy/falsy 依存でバグを生みやすい箇所を指摘する
- 共有ユーティリティ化が妥当な重複と、過剰抽象化を見分けてレビューする
- eslint-disable / ts-ignore / ts-expect-error を使う場合は、理由コメントの有無を確認する

## セキュリティ
- 外部入力・API レスポンス・環境変数は信頼せず、検証前提で扱う
- シリアライズ/デシリアライズ、HTML 挿入、動的評価などは安全性を確認する

## テスト
- バグ修正や型の穴埋めには、回帰テストや型テストの追加余地を確認する

## レビューの優先順位
- 指摘の優先順位は「正しさ > セキュリティ > 壊れやすさ > 保守性 > 性能 > スタイル」の順にする
- 単なる見た目や好みより、状態管理・副作用・責務分離・XSS を優先して見る
- アクセシビリティは現在の運用対象外のため、レビューで言及しない

## ディレクトリ構成 / コロケーション
- 機能単位でのコロケーションを優先する
- 早すぎる共通化は避け、軽微な重複よりも責務の近さと変更容易性を重視する
- コンポーネント、フック、型、ユーティリティ、API は、必要以上に広域共有せず近い責務の場所に置く
- 共有化は「複数箇所で本当に再利用される」「責務が安定している」場合に限って提案する
- 型定義は原則 `_types/` ディレクトリに配置する
- feature 内部でのみ使う型は `_types/` に置き、複数 feature で共有される型のみ共有ディレクトリに配置する
- 他機能の内部実装に直接依存していないか確認する
- 共有化が必要な場合のみ、最も近い共通の親ディレクトリに `_components`、`_hooks`、`_lib`、`_types`、`_utils`、`api` などを配置する
- ディレクトリ構造で機能の境界が読み取れるかを重視する

## import ルール
- `@/app/...` 形式の絶対パス import は避ける
- `@/lib/...` と `@/components/...` は、全体で共有すべき安定したリソースに限定して使う
- 認証・セッションなど横断的責務は例外として許容する
- 別セクションの内部 API や内部実装への密結合がないか確認する

## React パターン
- `useEffect` は本当に副作用かを確認し、計算で済むものを effect に入れない
- `useEffect` 内で参照する値が依存配列に正しく含まれているか確認する
- 非同期処理では race condition、二重実行、unmount 後更新を避けるため cleanup や AbortController を検討する
- state のリセットを effect に頼りすぎていないか確認し、必要なら key prop や state の持ち上げ/分離を検討する
- `useCallback` / `useMemo` は慣習で追加せず、再レンダリング抑制や依存安定化の必要がある場合のみ評価する
- 子へ渡す関数や重い計算以外で過剰最適化していないか確認する

## 型安全性
- `any` は使わない
- 不確実な値には `unknown` を使い、型ガード等で安全に絞り込む
- `eslint-disable` / `@ts-ignore` / `@ts-expect-error` には理由コメントがあるか確認する
- 未使用引数が必要な場合は `_` prefix を推奨する

## コメント
- コメントは全件必須ではなく、非自明な型、複雑な UI 状態、useEffect の意図、ワークアラウンドの理由などに絞る
- 「何をしているか」より「なぜそうしているか」を優先する

## UI / UX
- ローディング、エラー、空状態が考慮されているか確認する
- フォーム入力やボタン操作で無効化・多重送信防止・エラーメッセージが適切か確認する

## セキュリティ
- `dangerouslySetInnerHTML` の使用は XSS 観点で厳しく確認する
- API レスポンスにパスワード・トークン・内部情報など機密値を含めていないか確認する
- クライアントに出す必要のない情報をレスポンスに載せていないか確認する

## テスト
- 分岐の多い UI 状態やバグ修正には、表示・操作・副作用に対する回帰テスト追加を検討する

## 優先して確認すること
- 不要に広い `permissions` を与えていないか
- `pull_request_target` の利用が安全か
- untrusted input を shell にそのまま渡していないか
- `${{ }}` の展開に secrets や危険入力の漏洩リスクがないか
- 外部 Action は信頼できるか、可能ならバージョン pin や commit SHA pin があるか
- キャッシュキーや artifact に機密情報が混ざらないか
- fork 由来 PR で secrets が露出しない設計か
- `run:` スクリプトで `set -euo pipefail` 相当の安全性が必要か確認する
- 本番デプロイや destructive な処理に適切なガードがあるか
- 並列実行や再実行で事故る処理には concurrency 制御が必要か確認する
